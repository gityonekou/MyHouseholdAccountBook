## 1. エグゼクティブサマリー

マイ家計簿は、個人の家計管理を支援するSpring Bootベースのwebアプリケーションです。現在、DDDの要素を部分的に取り入れたレイヤードアーキテクチャで実装されていますが、ユースケース層に業務ロジックが集中しており、ドメインモデルの活用が不十分な状態です。

### 主要な発見事項

- **コード規模**: 44,719行のJavaコード（376ファイル）
- **アーキテクチャ**: レイヤードアーキテクチャ（4層）
- **主要な問題**: ユースケース層への業務ロジック集中
- **リファクタリング推奨度**: 高（DDD完全適用により保守性・拡張性が大幅に向上）

## 3. 現在のアーキテクチャ分析

### 3.1 レイヤー構成

現在のアプリケーションは、以下の4層アーキテクチャで構成されています：

```
┌─────────────────────────────────────┐
│   Presentation Layer                │
│   (Controller, Request, Response)   │
└─────────────────────────────────────┘
              ↓
┌─────────────────────────────────────┐
│   Application Layer                 │
│   (UseCase)                         │  ← ★業務ロジックが集中
└─────────────────────────────────────┘
              ↓
┌─────────────────────────────────────┐
│   Domain Layer                      │
│   (Model, Repository, Type)         │  ← ★活用不十分
└─────────────────────────────────────┘
              ↓
┌─────────────────────────────────────┐
│   Infrastructure Layer              │
│   (DataSource, DTO, Mapper)         │
└─────────────────────────────────────┘
```

## 4. 主要機能

### 4.1 機能一覧

アプリケーションは以下の主要機能を提供しています：

#### 収支管理機能
1. **月次収支照会** - 月ごとの収入・支出の表示
2. **年次収支照会** - 年間の収支サマリー表示
3. **収支登録** - 収入・支出の登録
4. **収支更新** - 既存データの更新

#### 買い物登録機能
1. **買い物登録（通常）** - 詳細な買い物情報の登録
2. **買い物登録（簡易）** - 簡易的な買い物情報の登録
3. **商品選択** - 登録済み商品からの選択登録

#### マスタ管理機能
1. **支出項目管理** - 支出カテゴリの管理
2. **店舗管理** - よく利用する店舗の管理
3. **商品管理** - 購入商品の管理
4. **固定費管理** - 定期的な固定費の管理
5. **イベント管理** - イベント関連支出の管理

#### システム管理機能
1. **ユーザー管理** - ユーザー情報の管理
2. **対象年月更新** - 現在の対象年月の管理
3. **基準情報管理** - システム基準データの管理

## 6. 現状の問題点分析

### 6.1 アーキテクチャ上の問題

#### 問題1: ユースケース層への業務ロジック集中

**現状**:
- ユースケースクラスが200行以上になっているケースが多数
- データ整合性チェックがユースケース層に実装されている
- ドメインモデルからレスポンスへの変換ロジックがユースケースに記述

**具体例**:
```java
// AccountMonthInquiryUseCase.java（269行）
private AccountMonthInquiryResponse execRead(...) {
    // データ取得
    AccountMonthInquiryExpenditureItemList resultList = repository.select(inquiryModel);
    
    // 整合性チェック（本来はドメインモデルで行うべき）
    if(!chkIncomeAmount.getSyuunyuuKingakuTotalAmount().equals(incomeKingakuTotalAmount)) {
        throw new MyHouseholdAccountBookRuntimeException("該当月の収入情報が一致しません...");
    }
    
    // 変換処理（本来はドメインモデルで行うべき）
    response.addExpenditureItemList(convertExpenditureItemList(resultList));
    ...
}
```

**影響**:
- テストが困難（ユースケース層のテストに業務ロジックのテストが含まれる）
- 業務ロジックの再利用が困難
- 保守性の低下

#### 問題2: ドメインモデルの貧弱性（貧血ドメインモデル）

**現状**:
- ドメインモデルは主にデータ保持のみ
- 業務ルールがドメインモデル外に散在
- ドメインサービスの不足

**具体例**:
```java
// IncomeAndExpenditureItem.java
// データ保持とファクトリメソッドのみで、業務ロジックが少ない
public class IncomeAndExpenditureItem {
    private final SyuunyuuKingaku syuunyuuKingaku;
    private final SisyutuKingaku sisyutuKingaku;
    // ... getterのみ
    
    // 業務ロジックがほとんどない
    public boolean isEmpty() {
        return userId == null;
    }
}
```

**影響**:
- ビジネスロジックとデータが分離され、凝集度が低い
- ドメイン知識がコード上で表現されていない
- 変更時の影響範囲が広い

#### 問題3: 集約の境界が不明確

**現状**:
- 集約ルート（Aggregate Root）が定義されていない
- 関連エンティティへのアクセス制御がない
- トランザクション境界が不明確

**影響**:
- データ不整合のリスク
- 並行処理時の問題
- ビジネス不変条件の維持が困難

#### 問題4: バリューオブジェクトの活用不足

**現状**:
- 値オブジェクトは定義されているが、単純なラッパーに留まる
- バリデーションロジックが値オブジェクト内に十分含まれていない
- 値オブジェクト間の計算ロジックがユースケース層に存在

**改善の余地**:
- 値オブジェクトにバリデーションを含める
- ドメイン計算ロジックを値オブジェクトに移動
- 不変性の徹底

### 6.2 コード品質上の問題

#### 問題5: ステップ数の多いクラス

**現状**:
- 一部のユースケースクラスが200行以上
- 責任が集中している

**推奨**:
- 単一責任の原則に基づく分割
- ドメインサービスへの抽出

#### 問題6: テストの難しさ

**現状**:
- ユースケース層に業務ロジックが集中しているため、テストが複雑
- モックが多くなり、テストの保守性が低い

**改善案**:
- ドメインモデルに業務ロジックを移動し、単体テストを容易に
- 統合テストとユニットテストの明確な分離

## 7. 良い点・評価できる点

現在の実装で評価できる点：

### 7.1 レイヤー分離

✅ **明確なレイヤー構造**
- Presentation、Application、Domain、Infrastructureの4層に分離
- パッケージ構造が整理されている

### 7.2 値オブジェクトの採用

✅ **型安全性の確保**
- UserId、TargetYearMonth、SyuunyuuKingakuなどの値オブジェクトを使用
- プリミティブ型の直接使用を避けている

### 7.3 リポジトリパターン

✅ **データアクセスの抽象化**
- リポジトリインターフェースをドメイン層に定義
- 実装をインフラストラクチャ層に配置

### 7.4 不変オブジェクト

✅ **安全性の高い設計**
- ドメインモデルを不変オブジェクトとして実装
- Lombokの@RequiredArgsConstructor(access = AccessLevel.PRIVATE)を活用

### 7.5 ファクトリメソッド

✅ **生成ロジックの集約**
- 静的ファクトリメソッド（from, create系）を適切に使用
- 生成ロジックがカプセル化されている

## 8. リファクタリング方針

### 8.1 基本方針

1. **段階的なリファクタリング**
   - 既存機能を壊さない
   - テストを書きながら進める
   - 1機能ずつ完全にDDD化

2. **ドメイン駆動設計の完全適用**
   - エンティティ、値オブジェクト、集約の明確化
   - ドメインサービスの抽出
   - ユビキタス言語の確立

3. **テスト戦略**
   - ドメインモデルの単体テスト充実
   - 統合テストの整備
   - テストカバレッジ80%以上を目標

### 8.2 優先順位

#### Phase 1: 基盤整備（高優先度）
1. ドメインモデルの強化
   - 集約の定義
   - ドメインサービスの抽出
   - エンティティと値オブジェクトの整理

2. 境界づけられたコンテキストの明確化
   - コンテキストマップの作成
   - ユビキタス言語の定義

#### Phase 2: ユースケースリファクタリング（中優先度）
1. 業務ロジックのドメイン層への移動
2. ユースケースの責任縮小
3. ドメインイベントの導入

#### Phase 3: インフラストラクチャ改善（低優先度）
1. リポジトリ実装の最適化
2. パフォーマンス改善

## 10. まとめ

マイ家計簿アプリケーションは、既に良好なレイヤードアーキテクチャを持ち、DDDの要素も部分的に取り入れられています。しかし、ユースケース層への業務ロジックの集中という課題があり、完全なDDD適用により以下のメリットが期待できます：

### 期待される効果

1. **保守性の向上**
   - 業務ロジックがドメインモデルに集約され、変更が容易に
   - 単一責任の原則により、影響範囲が限定的に

2. **テスト容易性の向上**
   - ドメインロジックの単体テストが容易に
   - モックの削減

3. **拡張性の向上**
   - 新機能追加時の影響範囲が明確に
   - ドメイン知識の再利用が容易に

4. **可読性の向上**
   - ビジネスロジックがドメインモデルに表現される
   - コードがドキュメントとして機能

### リファクタリング推奨度

**⭐⭐⭐⭐⭐ 強く推奨**

現在の実装は良好な基盤を持っており、DDD完全適用により大幅な品質向上が見込めます。
