## 支出項目マスタ（SISYUTU_ITEM_TABLE）

支出項目マスタは階層構造を持つマスタデータで、以下のルールに従って採番します。

### マスタデータの制約（重要）

**システム設計上の制約**：
- 支出項目コード0001〜0060は**固定値（マスタデータ）**として運用される
- これらのマスタデータは**削除・更新できない**ようにシステムが設計されている
- 運用開始後に追加される支出項目は0061以降のコードが割り当てられる

**買い物登録機能で参照される必須の支出項目コード**：

| 支出項目コード | 支出項目名 | 用途 |
|--------------|-----------|------|
| 0007 | 流動経費 | 仕事(流動経費)の買い物登録 |
| 0046 | 被服費 | 被服費の買い物登録 |
| 0047 | 住居設備 | 住居設備の買い物登録 |
| 0050 | 日用消耗品 | 日用消耗品の買い物登録 |
| 0051 | 食費 | 飲食の買い物登録 |
| 0052 | 一人プチ贅沢・外食 | 外食の買い物登録 |

**テストデータへの影響**：
- テストでは上記の必須支出項目コードが存在することを前提としたビジネスロジックがある
- `ShoppingRegistExpenditureAndSisyutuKingakuComponent`の`checkExpenditureRegistItemList`メソッドが参照
- テストデータには本番環境と同等のマスタデータ（0001〜0060）を登録する必要がある
- 本番環境のマスタデータ：`src/main/resources/sql/initsql/data.sql`

### SISYUTU_ITEM_CODE（支出項目コード）

**採番ルール**：
- レコードの登録順に連番で採番
- 親子関係は無関係
- 4桁の0パディング形式

**例**：
```sql
-- 1件目のレコード
SISYUTU_ITEM_CODE = '0001'

-- 2件目のレコード
SISYUTU_ITEM_CODE = '0002'

-- 3件目のレコード
SISYUTU_ITEM_CODE = '0003'
```

### SISYUTU_ITEM_SORT（支出項目ソート順）

**採番ルール**：
- 10桁の数値文字列で階層構造を表現
- 各階層で2桁ずつ使用し、親の階層の値を引き継ぐ
- 同一階層内では連番を割り当てる
- SISYUTU_ITEM_LEVELは1～5まで対応

**階層構造の例**：

```
Level 1（最上位階層）:
  [01][00][00][00][00] = 0100000000  ← 1件目
  [02][00][00][00][00] = 0200000000  ← 2件目
  [03][00][00][00][00] = 0300000000  ← 3件目

Level 2（Level 1の配下）:
  [01][01][00][00][00] = 0101000000  ← Level1の「01」配下の1件目
  [01][02][00][00][00] = 0102000000  ← Level1の「01」配下の2件目
  [01][03][00][00][00] = 0103000000  ← Level1の「01」配下の3件目

  [02][01][00][00][00] = 0201000000  ← Level1の「02」配下の1件目
  [02][02][00][00][00] = 0202000000  ← Level1の「02」配下の2件目

Level 3（Level 2の配下）:
  [01][02][01][00][00] = 0102010000  ← Level2の「0102」配下の1件目
  [01][02][02][00][00] = 0102020000  ← Level2の「0102」配下の2件目
  [01][02][03][00][00] = 0102030000  ← Level2の「0102」配下の3件目

Level 4（Level 3の配下）:
  [01][02][01][01][00] = 0102010100  ← Level3の「010201」配下の1件目
  [01][02][01][02][00] = 0102010200  ← Level3の「010201」配下の2件目

Level 5（Level 4の配下）:
  [01][02][01][01][01] = 0102010101  ← Level4の「01020101」配下の1件目
  [01][02][01][01][02] = 0102010102  ← Level4の「01020101」配下の2件目
```

### PARENT_SISYUTU_ITEM_CODE（親支出項目コード）

**設定ルール**：
- Level 1（最上位階層）の場合：**自分自身のSISYUTU_ITEM_CODE**を設定
- Level 2以降の場合：親レコードのSISYUTU_ITEM_CODEを設定

**重要な注意**：
- `'0000'`というコードは存在しない
- Level 1のレコードは自己参照となる（例：SISYUTU_ITEM_CODE='0001'の場合、PARENT_SISYUTU_ITEM_CODE='0001'）
- この設計により、子から親をたどる処理で終了条件を判定できる
- ※この設計の妥当性については、今後のリファクタリング時に再検討予定

### 具体的なテストデータ例

```sql
INSERT INTO SISYUTU_ITEM_TABLE (
    USER_ID, SISYUTU_ITEM_CODE, SISYUTU_ITEM_NAME,
    SISYUTU_ITEM_DETAIL_CONTEXT, PARENT_SISYUTU_ITEM_CODE,
    SISYUTU_ITEM_LEVEL, SISYUTU_ITEM_SORT, ENABLE_UPDATE_FLG
) VALUES
-- Level 1: 最上位階層（自己参照）
('user01', '0001', '食費', '食費関連', '0001', '1', '0100000000', false),
('user01', '0004', '日用品', '日用品関連', '0004', '1', '0200000000', false),
('user01', '0006', '趣味娯楽', '趣味娯楽関連', '0006', '1', '0300000000', false),

-- Level 2: 食費（0001）の配下
('user01', '0002', '食料品', '食料品購入', '0001', '2', '0101000000', false),
('user01', '0003', '外食', '外食費', '0001', '2', '0102000000', false),

-- Level 2: 日用品（0004）の配下
('user01', '0005', '消耗品', '日用消耗品', '0004', '2', '0201000000', false),

-- Level 2: 趣味娯楽（0006）の配下
('user01', '0007', '娯楽', '娯楽費', '0006', '2', '0301000000', false);
```

### 重要な注意点

1. **SISYUTU_ITEM_CODEは登録順の連番**
   - データ挿入順序が重要
   - 親→子の順序で挿入する必要はない（連番が優先）

2. **SISYUTU_ITEM_SORTは階層構造を反映**
   - 画面表示順序を制御するための項目
   - SQLのORDER BY句で使用される
   - 階層構造に従った論理的な順序を保持

3. **外部キー参照**
   - EXPENDITURE_TABLE、SISYUTU_KINGAKU_TABLEなどから参照される
   - テストデータ修正時は参照元も合わせて修正が必要

## その他のマスタデータ

### ACCOUNT_BOOK_USER（ユーザーマスタ）

**テストデータの基本方針**：
- USER_IDは'user01'、'user02'のような連番形式を使用
- NOW_TARGET_YEAR、NOW_TARGET_MONTHはテストシナリオに応じて設定

### INCOME_TABLE / EXPENDITURE_TABLE（収入・支出明細）

**コード採番ルール**：
- INCOME_CODE、EXPENDITURE_CODEは年月内での連番
- 3桁の0パディング形式（'001'、'002'、'003'...）

### SHOP_TABLE（店舗マスタ）

**コード採番ルール**：
- SHOP_CODEは3桁の0パディング形式
- SHOP_SORTは表示順序を制御（SHOP_CODEと同じ値を設定することが多い）
